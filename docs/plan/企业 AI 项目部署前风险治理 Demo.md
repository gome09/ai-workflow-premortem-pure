# 二、产品边界

## 2.1 正确定位

系统应帮助企业团队在 AI 应用投入试点或部署之前回答：

* 这个 AI 系统可能怎样失败？
* 哪些失败会影响用户、组织或数据安全？
* 哪些节点必须由人审核？
* 应如何验证风险控制是否有效？
* 当前项目是否可以试点、是否需要附加条件，或者应当停止部署？

## 2.2 非目标

不要声称或暗示系统能够完成：

* 完整企业项目立项；
* 投资回报和预算审批；
* 项目排期与资源配置；
* 采购和合同审批；
* 法律、医疗或监管意义上的正式认证；
* ISO 42001 合规认证；
* 生产系统最终验收；
* 通过 AI 自动批准高风险项目上线。

所有文档、界面和报告中出现的类似表述都要调整。

推荐统一使用以下表述：

> 本系统提供 AI 项目部署前的结构化风险预评估、人机监督设计、评测验证和门禁决策支持。最终业务责任和部署批准仍由授权人员承担。

# 三、必须保留并强化的四阶段流程

四个阶段应当形成明确的数据依赖关系，而不是四组互不相关的 LLM 输出。

## Stage 1：失败模式识别

目标：

> 基于项目目标、应用领域、目标用户、数据敏感性和使用边界，识别具有领域针对性的失败模式。

每条失败模式至少应包含：

* 稳定且唯一的 `failure_mode_id`；
* 风险类别；
* 具体失败描述；
* 受影响对象或利益相关者；
* 可能后果；
* 严重度；
* 发生可能性；
* 是否需要人工复核；
* 支持证据或依据；
* 建议控制方向；
* 尚待核实的问题。

要求：

1. 失败模式必须与当前场景语义一致。
2. 不得只输出“幻觉”“上下文遗忘”等泛化风险。
3. 高风险场景必须覆盖领域特有风险。
4. Stage 1 输出必须成为 Stage 2、Stage 3 和 Stage 4 的可追踪输入。
5. 若修改持久化模型，应同步更新 schema 版本、迁移和兼容逻辑。
6. `stages/schemas.py` 与 `core/models.py` 中重复的字段定义必须保持一致；允许进行有限重构，但不要展开与本任务无关的大规模模型拆分。

## Stage 2：人机监督设计

目标：

> 针对 Stage 1 的失败模式，在 AI 工作流中定义具体的人类责任、审核触发条件、决策权限和升级路径。

每个人机监督节点至少应明确：

* `node_id`；
* 节点业务目的；
* 对应的 `failure_mode_id` 列表；
* AI 在该节点能做什么；
* AI 明确不能做什么；
* 触发人工审核的条件；
* 执行人工审核的角色；
* 人工可以执行的动作；
* 审核标准；
* 所需证据；
* 是否允许自动继续；
* 升级条件；
* 高风险情况下的停止或回退动作。

必须实现以下一致性规则：

1. 每个 high 或 critical 失败模式必须至少映射到一个人工监督节点。
2. 每个监督节点引用的失败模式 ID 必须真实存在。
3. high 或 critical 节点默认不得自动继续。
4. 人工审核不能只写“请人工检查”，必须明确检查对象、标准、角色和处置动作。
5. 未覆盖的高风险失败模式应形成阻断型门禁项。
6. 人工编辑、批准、驳回、核验证据和升级动作应继续通过现有 `OversightService` 和确定性状态转移逻辑执行，不允许由 LLM 直接推进阶段。

## Stage 3：压力测试与评测

目标：

> 将失败模式和人机监督设计转换成可以执行或演示的测试用例，验证控制措施是否有效。

每个测试用例至少应包含：

* `case_id`；
* 对应的 `failure_mode_id`；
* 对应的 `node_id`；
* 场景类型：normal、edge 或 adversarial；
* 测试输入；
* 预期安全行为；
* 禁止出现的行为；
* 可验证的通过标准；
* 实际结果；
* 证据类型；
* 是否为 Mock 演示证据；
* 人工评审结果；
* 最终通过状态。

重点修复：

1. 不得仅依赖 fixture 中预先写死的 `passed: true` 或 `overall_passed: true`。
2. `overall_passed` 应由测试结果、门禁条件和人工审核状态确定性计算。
3. Mock 模式可以使用确定性演示结果，但必须明确标记为：

   * demo evidence；
   * 非生产运行证据；
   * 非独立验收结果。
4. 每个 high 或 critical 失败模式必须有对应测试覆盖。
5. 每个关键人工监督节点至少有一个正常或边界测试。
6. 对抗性风险应通过 Red Team 或 adversarial EvalCase 覆盖。
7. 测试失败后必须产生明确的整改、回退或阻断结果。
8. Stage 3 不能只“生成测试”，还应展示测试结果如何影响部署决策。

## Stage 4：部署前门禁与决策建议

保持 Stage 4，不新增业务阶段，但将其语义从简单的“触发方式”强化为：

> 综合失败模式、人机监督覆盖、评测结果、未关闭风险和人工审核状态，形成部署前门禁结论。

Stage 4 应保留现有 `trigger_methods` 的兼容能力，同时增加清晰的部署决策结构。

建议增加类似以下模型：

* `decision`：

  * `go`
  * `conditional_go`
  * `pilot_only`
  * `no_go`
* `decision_scope`：

  * 仅内部测试；
  * 受限试点；
  * 有条件部署；
  * 暂停部署。
* `decision_rationale`
* `unresolved_risk_ids`
* `required_conditions`
* `required_approvals`
* `monitoring_requirements`
* `rollback_conditions`
* `prohibited_uses`
* `review_after`
* `human_accountable_role`
* `is_demo_recommendation`

必须满足：

1. 存在未关闭的 critical 风险时不得给出 `go`。
2. 存在未解决的 high 风险、阻断动作或失败评测时不得给出无条件 `go`。
3. `conditional_go` 必须列出可验证的前置条件。
4. `pilot_only` 必须明确试点范围、人工监督和停止条件。
5. `no_go` 是合法且完整的评估结论，不应被视为系统执行失败。
6. 工作流的 `complete` 只能表示“本次部署前风险评估已经完成”，不能表示“生产部署已获正式批准”。
7. 最终报告和 UI 必须明确区分：

   * 评估流程完成；
   * 门禁允许继续；
   * 企业授权人员正式批准部署。
8. Stage 4 决策必须由确定性规则校验，不能完全相信 LLM 自报结论。

* AI 不替代专业心理咨询人员；
* 危机信号必须升级给授权专业人员；
* 高风险输出必须人工复核；

* 此场景允许最终得到 `pilot_only`、`conditional_go` 或 `no_go`；
* 不得输出以“学术诚信”“教学效果”为主的风险内容。

# 五、修复风险分级逻辑

重点检查：

* `core/gates/risk_profile.py`
* 相关风险分类和门禁测试。

当前风险关键词逻辑不得允许以下错误：

> 高影响领域因为文本中出现“辅助”“学习”“建议”“测试”“Demo”等低风险词，被从 HIGH 降为 MEDIUM 或 LOW。

请采用“领域风险下限”或等价的单调风险规则。

最低要求：

1. 医疗、药物、临床诊断场景风险下限为 CRITICAL，除非项目明确只是在分析公开非敏感文本且代码规则能够证明例外合理。
2. 心理健康、自伤、危机场景风险下限至少为 HIGH。
3. 涉及敏感个人数据时风险下限至少为 HIGH。
4. 涉及未成年人、高影响自动决策时不得因低风险用途词被降级。
5. Stage 1 新生成的文本只能维持或提高已有领域风险，不得意外降低初始风险。
6. “Demo”“测试”“辅助”等词可以影响部署范围，但不能覆盖领域风险下限。
7. 分类结果必须提供可读理由，UI 和报告可以展示这些理由。

增加属性或回归测试，至少验证：

* 心理健康场景在 Stage 1 前后均不低于 HIGH；
* “心理健康辅助学习 Demo”仍不低于 HIGH；
* 医疗诊断原型仍为 CRITICAL；
* 通用个人笔记助手可以是 LOW；
* 普通企业 RAG 默认可为 MEDIUM；
* 高风险自动化可提高风险等级；
* 低风险词不会覆盖高风险领域下限。

# 九、测试要求

不得只修改现有断言以迎合新输出。必须增加能够发现业务语义错误的测试。

至少增加以下测试类别。

## 9.1 场景语义测试

验证每个场景包含指定关键词或风险类别，并排除明显错位内容。

示例：

* 心理健康场景必须包含危机升级、隐私、漏报或误报等领域风险；
* 心理健康场景不得以学术诚信作为主要风险；
* 学生选课场景必须包含先修条件、时间冲突、毕业要求中的至少若干项；
* 通用 RAG 场景必须包含无依据回答、权限或 Prompt Injection 风险；
* 每个 scenario manifest 必须指向匹配自身的 fixture。

不要只检查中文字符串。优先检查结构化 category、risk code、failure mode ID 和映射关系。

## 9.2 风险等级测试

覆盖前述风险下限和单调性规则。

## 9.3 跨阶段合同测试

验证：

* Stage 1 → Stage 2 引用完整；
* Stage 2 → Stage 3 引用完整；
* Stage 3 → Stage 4 决策一致；
* high/critical 风险有人工节点和测试覆盖；
* 失效引用能够阻断；
* 未解决阻断项时不能产生 `go`。

## 9.4 Stage 3 结果计算测试

验证：

* `overall_passed` 由测试结果确定；
* 任一阻断型失败用例会影响整体结果；
* Mock 结果带有 demo evidence 标识；
* 人工未复核时，高风险测试不得视为最终通过。

## 9.5 Stage 4 决策测试

至少验证：

* critical open risk → 不得 `go`；
* high open risk → 不得无条件 `go`；
* failed eval → `no_go` 或 `conditional_go`；
* 全部门禁通过且无阻断 → 可 `pilot_only`、`conditional_go` 或 `go`；
* `conditional_go` 必须有 conditions；
* `pilot_only` 必须有 scope 和 stop/rollback conditions；
* 评估完成不等于正式部署批准。

## 9.6 E2E 场景测试

为四个内置场景分别增加端到端测试，验证：

* 能创建会话；
* 能完成预期的四阶段交互或得到预期阻断；
* 阶段输出语义一致；
* 人工动作可处理；
* 最终报告可以导出；
* 最终决策与风险等级一致。

高风险场景可以以阻断、专家升级或 `no_go` 作为成功的 E2E 结果。不要强迫所有场景最终显示绿色通过。
