# v0.7.0-alpha.4 Stage Advancement Hardening Plan

Status: source-level implementation patch; runtime validation deferred by instruction.

## Scope

This patch implements the next-stage strategy without redesigning the project and without starting v0.8/v0.9 work.

Implemented scope:

1. `StageAdvancementDecision` as the single stage-advance read model.
2. `stage_advancement_coordinator` to compose gate, resolution operations, action sync, stage mutation summaries, and trace records.
3. Stage advance API:
   - `GET /sessions/{session_id}/stages/{stage_id}/advancement-decision`
   - `POST /sessions/{session_id}/stages/{stage_id}/advance`
4. Graph review approval now calls the same coordinator instead of independently interpreting `StageGateResult`.
5. Action/evidence/safety/stage-operation paths now refresh and/or record a stage advancement decision.
6. Remaining GateRule wrappers were migrated to direct rule classes while keeping legacy collectors in `stage_readiness_service.py` for compatibility.
7. Frontend client/components can display the unified decision and operation list.
8. Trace typing was extended to `action` and `stage_operation`.

## Advancement contract

A stage can advance only when:

```text
StageAdvancementDecision.can_advance == true
```

A stage was actually advanced only when:

```text
StageAdvancementDecision.advanced == true
```

Read-only callers should use the GET endpoint. Mutating callers should use the POST `/advance` endpoint or graph review approval, both of which call the same coordinator.

## Validation boundary

This package intentionally contains code and test fixtures only. It has not run pytest or service startup checks. Unified validation should be performed after all planned hardening changes are complete.
